# LINSTOR — Missing `/dev/linstor_vg/*` symlinks after node reboot

**Status:** Active issue, no upstream fix. Mitigated by holding all kured-driven node reboots while LINSTOR is in use.
**First observed:** 2026-04-14, on opi02 and opi03 after opi03's auto-upgrade reboot at 03:38 UTC.
**Time-to-diagnose on first occurrence:** ~4 hours of misdirected debugging across DRBD, ArgoCD, VolSync, and snapshot subsystems.
**Related:**
- [piraeus-snapshot-consistency.md](../architecture/storage/piraeus-snapshot-consistency.md)
- [linstor-volsync-resize-inode-corruption.md](./linstor-volsync-resize-inode-corruption.md)

---

## TL;DR

After a node reboot, `udev` sometimes silently fails to create LVM device-mapper symlinks at `/dev/linstor_vg/<pvc-uuid>_00000` for some subset of LINSTOR-managed LVs. The LV exists in LVM metadata (`lvs` reports it), the device-mapper device exists in the kernel (`dmsetup ls` reports it), but the path symlink that LINSTOR's satellite (and consequently every snapshot/mount operation) needs is **gone**.

Failures cascade in confusing directions: VolSync mover pods stuck, snapshot creation timeouts, false "DRBD split-brain" reports, ArgoCD sync immutability errors, missing cache PVCs. None of those symptoms point at the actual cause.

**The fix is `sudo vgmknodes linstor_vg` on every affected satellite node.** The fix is 10 seconds. The discovery cost on first occurrence is hours.

---

## Symptoms

You'll see some combination of these, often all at once, and they look unrelated:

### From LINSTOR

When you try to take a snapshot via `linstor snapshot create` or via Kubernetes `VolumeSnapshot`, the command fails with:

```
ERROR:
Description:
    (opi0X) No response generated by handler.
Details:
    In API call 'ChangedData'.
```

The `linstor node list` shows all satellites `Online`. `linstor resource list` shows the resource as `Ok` and `UpToDate` everywhere. There's no DRBD-level error visible, just a satellite that "doesn't respond" during the snapshot's distributed coordination phase.

### From the satellite logs

`kubectl logs -n piraeus-system linstor-satellite.opi0X-XXXXX -c linstor-satellite` will show:

```
[DeviceManager] ERROR LINSTOR/Satellite/XXXXXX SYSTEM - Failed to get block size of
/dev/linstor_vg/pvc-XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX_00000 [Report number ...]
```

The satellite is trying to read the LV's block size to align it for the snapshot. Reading fails because the path doesn't exist.

### From the host

```bash
$ sudo blockdev --getsize64 /dev/linstor_vg/pvc-XXXX..._00000
blockdev: cannot open /dev/linstor_vg/pvc-XXXX..._00000: No such file or directory

$ sudo lvs linstor_vg/pvc-XXXX..._00000
  LV                                             VG         Attr       LSize    ...
  pvc-XXXX..._00000                              linstor_vg Vwi-aotz-- 104.00m  ...
  # The LV exists in LVM metadata.

$ sudo dmsetup ls | grep XXXX
linstor_vg-pvc--XXXX--..._00000  (253:11)
  # The DM device exists in the kernel — only the /dev path symlink is missing.
```

### Cascading downstream symptoms

Because the root cause is silent and only manifests when something tries to *use* the missing path, you'll typically discover it via the downstream effects first:

- **VolSync mover pods stuck in `ContainerCreating` or `Pending`** for hours, sometimes with `VolumeForcingIoErrors` events from the LINSTOR HA controller
- **Snapshot creation timeouts** (`rpc error: code = DeadlineExceeded`) on `VolumeSnapshot` resources
- **Apparent DRBD split-brain** where some peers show `StandAlone` connections — DRBD's coordination retries fail because the satellite can't complete the snapshot quiesce step, so connections get torn down and re-established repeatedly
- **ArgoCD `Failed` syncs** on apps whose PVCs are involved, with confusing `spec is immutable` errors as the controller reconciles state changes triggered by the failures
- **Missing cache PVCs** — VolSync controller logs show the controller stuck "deleting temporary objects" and never spawning new movers
- **`volsync_volume_out_of_sync == 1`** firing on multiple ReplicationSources

If you see two or more of these together, especially shortly after a node reboot, **check the symlinks first** before chasing any of the apparent causes.

## Diagnosis

```bash
# 1. List recently rebooted satellite nodes:
ssh opi01.internal 'uptime'
ssh opi02.internal 'uptime'
ssh opi03.internal 'uptime'
# (and any other LINSTOR satellites)

# 2. On each satellite, run vgmknodes — it reports any missing symlinks
#    AND fixes them in one pass:
ssh opi02.internal 'sudo vgmknodes linstor_vg'
# Example output of a node with the bug:
#   The link /dev/linstor_vg/pvc-3ab09213-..._00000 should have been created
#   by udev but it was not found. Falling back to direct link creation.
#   The link /dev/linstor_vg/pvc-d4a3631e-..._00000 should have been created ...
#   The link /dev/linstor_vg/pvc-d2805d3d-..._00000 should have been created ...
#   ... (one line per missing symlink)
#
# A clean node prints nothing.

# 3. Repeat on every satellite. The bug can affect any subset of satellites
#    and any subset of LVs on each — there's no obvious pattern. opi01 may
#    be clean while opi02 has 5 missing and opi03 has 3.
```

**Important:** `vgmknodes` is "lazy" — on a single run it only repairs symlinks it currently sees as missing. We've observed cases where the first pass fixed 5 LVs on a node, then a few minutes later the LINSTOR controller still complained about a 6th LV on the same node, and a second `vgmknodes` pass fixed it. Treat the first pass as necessary-but-not-sufficient: re-run it whenever a new "no response generated by handler" error appears.

## Fix

```bash
# Per affected satellite — root cause fix is a single command:
ssh opi0X.internal 'sudo vgmknodes linstor_vg'
```

After the symlinks are restored, no further intervention is needed at the LINSTOR level — subsequent snapshot operations will succeed immediately. You may need to clean up the cascade fallout though:

```bash
# Stale failed LINSTOR snapshots from the period before the fix:
kubectl exec -n piraeus-system deploy/linstor-controller -- \
  linstor snapshot list 2>&1 | grep -E 'Failed|DELETING'
kubectl exec -n piraeus-system deploy/linstor-controller -- \
  linstor snapshot delete <resource-uuid> <snapshot-name>

# Stuck Kubernetes VolumeSnapshots:
kubectl get volumesnapshot -A | grep -E 'volsync.*-src.*false'
kubectl delete volumesnapshot -n <ns> <name>

# Stuck volsync mover pods (force delete if Terminating won't complete):
kubectl get pods -A -l app.kubernetes.io/created-by=volsync
kubectl delete pod -n <ns> <name> --force --grace-period=0

# Trigger a fresh manual sync to confirm the chain works end-to-end:
kubectl patch replicationsource -n <ns> <name> --type merge \
  -p "{\"spec\":{\"trigger\":{\"manual\":\"recover-$(date +%s)\"}}}"
# After it completes, drop the manual trigger so the schedule resumes:
kubectl patch replicationsource -n <ns> <name> --type json \
  -p '[{"op":"remove","path":"/spec/trigger/manual"}]'
```

## Why this happens (current understanding)

We don't have a confirmed root cause inside udev or LVM. What we know:

- `lvm2` ships udev rules that, on `change`/`add` events for `dm-N` devices, create the `/dev/<vg>/<lv>` symlinks via `13-dm-disk.rules` and friends.
- On a healthy node, those rules fire whenever LVM activates an LV.
- Something during the **post-reboot satellite reconnect cycle** appears to suppress, lose, or race with these udev events. The DM device gets created by LINSTOR (because `dmsetup ls` shows it), but udev never publishes the `/dev/linstor_vg/...` symlink for it.
- It's not all LVs — only a subset, and the subset is different per node and per reboot.
- `udevadm trigger` after the fact does not always re-fire the rules for these specific devices, but `vgmknodes` always works (because it bypasses udev entirely and creates the symlinks directly via a fallback path inside lvm2 — note its output says "Falling back to direct link creation").

Possible contributors (none verified):

1. **Memory/IO pressure during boot.** opi01 was at 14Gi/15Gi RAM the morning we discovered this; opi02 and opi03 are similarly close to the line under load. Heavy concurrent udev workers + LINSTOR satellite startup may starve udev for resources at the wrong moment.
2. **systemd / udev unit ordering with the LINSTOR satellite container.** The satellite runs as a Kubernetes pod and may be importing/activating LVs before udevd is fully ready, causing a race where the LV is activated but the udev event for it gets dropped.
3. **Aarch64 / Orange Pi kernel quirk.** All affected nodes are RK3588 (opi01-03 are Orange Pi 5 Plus). We haven't seen the bug on x86 nodes, but x86 is also a small sample (cardinal, hawk, routy don't run LINSTOR satellites).
4. **kernel 6.18.x udev/dm event handling regression.** opi03 went from 6.18.21 to 6.18.22 in the reboot that triggered the first occurrence. We don't have evidence either way.

If we ever investigate further, the key questions are:
- Does it reproduce on a manual reboot of opi02 or opi03 (without going through autoUpgrade)?
- Does it reproduce on x86 LINSTOR satellites?
- Is there a pattern in *which* LVs lose their symlinks (e.g., recently-modified, recently-used, oldest)?
- What does `journalctl -b 0 -u systemd-udevd` say around the LV activation timestamps?

## Mitigation in place

Two things are in place to prevent unplanned reoccurrence:

### 1. kured is disabled

`kubernetes/base/infra/system/kured/helm-values.yaml` pins the kured DaemonSet to a non-existent `nodeSelector`, so the cluster's automatic reboot orchestrator runs on zero nodes. NixOS `system.autoUpgrade` still builds new generations, but the reboot to apply them is held until a human pulls the trigger.

This is intentionally **all-or-nothing** rather than per-node: any reboot risks the bug on any satellite, so we hold all of them. To re-enable kured, drop the `nodeSelector` block from that helm-values file.

### 2. NixOS satellites set `system.autoUpgrade.allowReboot = false`

opi01, opi02, opi03 (and the LINSTOR-eligible raccoon nodes) all already set this, so NixOS itself never reboots them. The 2026-04-14 reboot came from kured, not from autoUpgrade. With kured disabled, both layers are now off.

## Manual reboot procedure (when you do need to reboot a satellite)

Until the underlying udev issue is fixed, treat every satellite reboot as a careful, supervised operation:

1. **One node at a time.** Don't reboot two satellites in the same window.
2. **Check VolSync state is healthy first.** No stuck movers, no `out_of_sync == 1` metrics, all `lastSyncTime` recent. Don't reboot into an existing problem.
3. **Drain the node** (or accept that pods using diskless LINSTOR clients on that node will reschedule).
4. **Reboot.**
5. **As soon as the node is back, run `sudo vgmknodes linstor_vg`** on it. Capture the output. If it reports any missing symlinks, that's confirmation the bug hit you again — file the output here so we accumulate data points.
6. **Take a test snapshot** of any LV that lives on the rebooted node:
   ```bash
   kubectl exec -n piraeus-system deploy/linstor-controller -- \
     linstor snapshot create <pvc-uuid> reboot-test-$(date +%s)
   kubectl exec -n piraeus-system deploy/linstor-controller -- \
     linstor snapshot list -r <pvc-uuid> | tail
   # Should show State=Successful with a real CreatedOn timestamp.
   # If it shows Failed, run vgmknodes again — there may be more missing symlinks.
   ```
7. **Then uncordon and proceed to the next node.**

---

### 3. Boot-time `vgmknodes` fixup (active since 2026-04-15)

`nixos/modules/nixos/services/k3s/default.nix` defines a `linstor-vgmknodes-fixup` systemd oneshot that runs on every boot of any host with `linstorSupport = true`:

- Ordered `after = linstor-loop-setup.service`, so the VG is activated before we run.
- Ordered `before = k3s.service`, so the symlinks are reconciled before the Piraeus satellite pod starts and queries the storage pool.
- Guarded with `vgs linstor_vg` so it's a no-op on hosts that don't have the VG (raccoon workers — they enable `linstorSupport` for the kernel modules but have no loop file).

This is **defense-in-depth, not a replacement for the kured/autoUpgrade hold**. The fixup only catches the boot-time class of the failure; mitigations 1 and 2 stay in place because:

- We still don't understand the root cause.
- We have no fixup for **late-activated LVs** (a PVC created post-boot, then losing its symlink some time later). The doc's "second-pass" observation in §Diagnosis is exactly this case. If we ever see evidence of late drops, the next step is a periodic systemd timer running the same script every ~15 min, plus a textfile-collector exporter to alert. **Not done yet — waiting for evidence.**

### Future: textfile-collector alerting (deferred)

A node-exporter textfile-collector running `vgmknodes --noudevsync` periodically and counting "Falling back to direct link creation" lines would give us proactive alerting for the late-drop case. Captured here for whoever picks it up if the boot-time fixup proves insufficient.
